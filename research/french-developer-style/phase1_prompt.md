# Research title

Find proven editorial rules for natural, clear French developer documentation and UI text generated by LLMs

# Goal

Run a broad discovery pass to find candidate sources that can help create an LLM skill for writing and editing French software-facing prose.

The target is not marketing copy, fiction, SEO text, or generic “humanisation”.
The target is clear, natural, technically precise French for software engineering contexts:

- technical documentation
- README files
- API docs
- code comments
- Javadoc/KDoc/docstrings translated or written in French
- changelog and release notes
- PR descriptions and review replies
- UI strings and UI translations
- error messages
- human-readable log messages

The practical goal is to avoid French LLM prose that sounds artificial, over-translated from English, bureaucratic, too verbose, or machine-generated.

Examples of possible failure modes to investigate:

- literal translations from English
- inconsistent terminology
- inconsistent formal/informal address
- awkward use of “vous”, “nous”, passive, or impersonal constructions
- excessive nominalizations
- long noun phrases
- overuse of “il est important de noter que”, “en conclusion”, “ainsi”, “de plus”, “permet de”
- false friends and anglicisms
- English-like sentence rhythm
- unnatural UI labels or button text
- awkward error messages
- inconsistent capitalization in headings and UI labels
- incorrect spacing before French punctuation
- incorrect quotation marks
- prose that sounds like generic corporate boilerplate rather than useful documentation

This is Phase 1 only. The goal is discovery and classification, not final skill authoring.

# Important framing

Do not look only for ready-made LLM skills. Ready-made French skills may not exist.

Search for reusable rule sources that could later be converted into a compact `SKILL.md`.

It is acceptable to conclude that no single proven French LLM skill exists. In that case, identify the strongest reusable rule sources and explain how they could later be combined.

Do not recommend AI-humanizer tools, detector-evasion prompts, paraphrasers, SEO rewriting tools, or generic copywriting prompts.

The goal is better reader experience:

- clarity
- technical precision
- natural French
- maintainability
- consistent terminology
- useful UI wording
- reduced native-speaker rejection

# French variant

French for France is the primary target unless a better source exists for another variant.

However, also include strong sources for Canadian French / Québec French, Swiss French, or international French if they are authoritative or especially relevant to software localization.

For each candidate, classify variant applicability as one of:

- France French
- Canadian French / Québec French
- Swiss French
- international French
- configurable / supports multiple variants
- unclear

If France French and Canadian French guidance differ materially, surface the difference. The final recommendation may involve:

- one France-French-first skill
- separate France and Québec variants
- one skill with a locale switch
- a core skill plus locale-specific terminology and typography modules

# Search scope

Search broadly across these categories.

## 1. French software documentation style guides

Look for public style guides from software companies, OSS projects, documentation teams, technical writing teams, and developer platforms.

Prioritize sources that cover:

- developer documentation
- API docs
- README files
- tutorials
- release notes
- error messages
- code comments
- terminology
- writing for developers in French

## 2. French UI and localization style guides

Look for French localization guidance from:

- Microsoft
- Apple
- Google
- Mozilla
- GitLab
- WordPress
- GNOME / KDE / Fedora / Debian / Ubuntu localization teams
- Transifex / Crowdin / Weblate communities
- Québec / Canadian government localization or terminology sources
- French government digital service sources if relevant
- major French-speaking software or SaaS companies

Focus on:

- UI strings
- button labels
- menu items
- placeholders
- tooltips
- error messages
- warnings
- plural forms
- placeholders and variables
- tone: “vous”, “tu”, impersonal, infinitive
- terminology consistency
- avoiding literal translation from English
- capitalization
- punctuation and spacing

## 3. French editorial and plain-language sources

Look for reputable sources on clear French prose, including:

- French editing guides
- technical writing guidance
- plain-language guidance
- style guides from editorial teams
- official terminology bodies
- public rule catalogues by named editors, translators, technical writers, or localization teams

Do not over-prioritize journalism or marketing style unless the rules clearly transfer to technical prose.

Useful categories may include:

- plain French / langage clair
- government digital writing guides
- Académie française / Office québécois de la langue française / TERMIUM Plus / FranceTerme
- guides on anglicisms, false friends, typography, and inclusive writing
- guides for writing procedures and help content

## 4. Linters, dictionaries, and mechanical checks

Look for tools and rule sets for French prose, spelling, grammar, typographic conventions, and terminology consistency:

- LanguageTool French
- Grammalecte
- Vale packages for French if any
- textlint plugins for French if any
- Antidote-related public guidance if available
- Hunspell dictionaries
- typographic tools for French punctuation spacing
- terminology checkers
- translation QA tools
- ICU MessageFormat / Fluent plural and gender handling guidance

Classify which rules belong in a linter rather than in an LLM skill.

## 5. Research and serious analysis of French LLM prose

Look for academic papers, editor blog posts, technical-writing discussions, and serious comparisons that identify French LLM stylistic tells.

Use these only to identify failure modes.
Do not optimize for AI-detector evasion.

If French-specific LLM-prose research is weak, identify transferable findings from multilingual or English research, but clearly mark them as indirect.

## 6. Agent skills, prompts, and rule packs

Look for:

- Claude / agent `SKILL.md` files
- Cursor / Continue rules
- public prompt libraries
- French technical-writing prompts
- French editing prompts
- localization-agent prompts

Treat these as candidates only if they show editorial quality, source reputation, or real-world use.

# What counts as evidence

Treat these as scoring signals, not hard filters:

- authored by a named editor, translator, technical writer, localization team, or documentation team
- adopted by a known software company, government digital service, or large OSS project
- used in a real documentation or localization workflow
- actively maintained
- includes concrete examples
- includes terminology decisions
- has public issue discussions or usage notes
- has a rule set that can be converted into `SKILL.md`
- has a linter/config that can be reused
- distinguishes French locales or provides locale-specific terminology

High star counts, install counts, and downloads are useful supporting evidence, but they must not exclude lower-profile sources with stronger editorial or localization authority.

# Exclusions

Exclude:

- generic marketing-copy prompts
- fiction-writing prompts
- SEO rewriting prompts
- “AI humanizer” tools
- paraphrasers aimed at bypassing AI detectors
- anonymous prompt snippets with no evidence of editorial quality
- grammar-only tools with no style or software-text relevance
- sources that only teach French grammar to learners
- sources focused only on academic writing unless directly useful for technical docs
- sources focused primarily on political debates around inclusive writing unless they give practical software-writing rules

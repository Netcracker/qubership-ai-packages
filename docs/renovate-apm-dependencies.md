# Renovate APM dependency updates

Renovate tracks APM package references in `apm.yml` files through the shared
[`Netcracker/renovate-config:apm`](https://github.com/Netcracker/renovate-config/blob/main/apm.json) preset. The
repository-local `renovate.json` enables that preset and disables automerge for dependencies extracted as `depType:
apm`.

## What Renovate updates

The shared preset scans every `apm.yml` file and handles these APM reference formats:

- Package dependencies pinned to release tags, such as
  `Netcracker/qubership-ai-packages/agent-packages/codex-review@v1.0.0`.
- Package dependencies pinned to immutable commits with a tracked branch or release-tag comment, such as
  `Netcracker/qubership-ai-packages/agent-packages/codex-review#<sha>  # main` or
  `Netcracker/qubership-ai-agent-telemetry/agent-packages/ai-agent-telemetry#<sha>  # v0.2.0`.
- Package dependencies that still reference a mutable branch directly, such as
  `Netcracker/qubership-core-lib-go/logging/agent-packages/logging-go-usage#feat/agent-packages`.
- Marketplace entries that use `source`, `subdir`, and `ref` fields in the root `apm.yml`.

Renovate groups these updates into an `apm packages` PR. For mutable branch references, Renovate pins the dependency to
the latest commit SHA and keeps the source ref as a trailing comment.

## Why APM updates do not automerge

The root `apm.yml` is the source of truth for marketplace packages, but
[`.claude-plugin/marketplace.json`](../.claude-plugin/marketplace.json) is generated by `apm pack`. Mend-hosted
Renovate cannot run repository-local `postUpgradeTasks`, so it cannot regenerate the marketplace index in the PR.

For that reason, repository-local `renovate.json` sets `automerge: false` for `depType: apm`. APM update PRs must stay
manual even when the change looks like a low-risk `digest` update. The marketplace validation workflow also runs
`apm pack --check-clean --dry-run` and fails when `apm.yml` and the committed marketplace index drift.

The first marketplace validation run on an APM Renovate PR is therefore expected to fail when Renovate changes the
root `apm.yml`. The `validate` job reports the marketplace drift, and `Marketplace Gate` propagates that failure until
the generated index is committed.

## How to finish a Renovate APM PR

1. Review the changed APM references in `apm.yml` files.
1. Ignore unrelated generated or documentation changes when comparing against a manually prepared update PR.
1. Regenerate the marketplace index:

   ```bash
   apm pack
   ```

1. Commit the regenerated `.claude-plugin/marketplace.json` with the Renovate branch.
1. Let the marketplace validation workflow pass before merging.

If Renovate updates only nested package manifests and not the root marketplace `apm.yml`, still run the validation
workflow. Package version bumps may require a matching root marketplace version entry before the release can pass
`apm pack --check-versions`.
